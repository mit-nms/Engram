## System Model

Here is how the broadcast routing works:

1. The system broadcasts data from a source node to multiple destination nodes across cloud networks.
  - The source node needs to send data to all destination nodes.
  - Data is split into `num_partitions` partitions (stripes) that can be routed independently.
  - Each partition must reach every destination node.

2. The network is represented as a directed graph (G) with cloud regions as nodes.
  - Nodes are formatted as `provider:region` (e.g., "aws:us-east-1", "gcp:europe-west1").
  - Each edge has two key attributes:
    - `cost`: Cost per GB to transfer data along this edge ($/GB).
    - `throughput`: Maximum data transfer rate (Gbps).
  - The graph contains 71 nodes across AWS, GCP, and Azure providers.

3. Each partition can be routed along different paths to reach each destination.
  - A partition's path from source to a destination is a sequence of edges.
  - Each edge tracks which partitions use it.

4. The system enforces ingress and egress limits per cloud provider.
  - Ingress limit: Maximum incoming bandwidth per provider (Gbps). These values are given in the graph G.
  - Egress limit: Maximum outgoing bandwidth per provider (Gbps). These values are given in the graph G.
  - The limits can not be exceeded.
  - Default limits: AWS ingress=10, egress=5; GCP ingress=16, egress=7; Azure ingress=16, egress=16 (per VM).

5. Transfer time is calculated based on bottleneck edges.
  - For each destination, all partitions must complete transfer.
  - Transfer time for a partition = max(time across all edges in its path).
  - Edge transfer time = (number of partitions using edge) × (partition size in GB) × 8 / (edge flow rate in Gbps).
  - Destination completion time = max(time across all its partitions).
  - Overall transfer time = max(time across all destinations).

6. Cost includes egress costs and instance costs.
   - Egress cost: Sum of (number of partitions using edge) × (partition size in GB) × (edge cost per GB) for all edges.
   - Instance cost: VM costs for nodes that process data (typically charged per second based on transfer time).
   - Total cost = egress cost + instance cost.

7. Problem parameters (used by simulator):
   - Transfer size: Typically 300 GB total (split across partitions)
   - Time budget: The simulator calculates transfer time based on bottlenecks
   - Partition size: TRANSFER_SIZE / num_partitions (typically 300 / num_partitions GB)
   - VM cost: Typically ~0.00015 $/second (0.54 $/hour)
   - VM limit: Typically 2 VMs per region (provides egress_limit * 2 Gbps capacity)

## Debugging Tips

When your MILP isn't working:

1. **Check solver status**: `prob.status` tells you if it's Optimal, Infeasible, or timed out
2. **Print constraint violations**: Add slack variables to identify which constraints are too tight
3. **Try to approximate with relaxations**
4. **Add time limits**: `PULP_CBC_CMD(timeLimit=60)` prevents infinite solving
5. **Verify the solution**: Check that the paths you build actually satisfy the constraints
6. **Use approximation as fallback, but log it**:
   ```python
   if prob.status != LpStatusOptimal:
   ```
